Thank you for trying GtfsProc! This is a GTFS data processor to present stop, route, service, and schedule lookups, but its principal purpose is to provide wait time estimations for stops in a transit system, integrating GTFS-Realtime data where available. All data processing is done on a server process in response to client queries over a TCP connection. This allows the process to work over network connections or on local machines.

GtfsProc was written almost entirely in the Qt Application Framework. The TCP server code comes from Bryan Cairns of VoidRealms. See http://www.voidrealms.com/ for the original source and license. The GtfsProc code is written by Daniel Brook (danb358 {AT} gmail {DOT} com), (c) 2018-2023, GPL v 3.

This is the documentation for GtfsProc Version 2.1.0 (02-Aug-2023)

Server Options

The GtfsProc server can be started with a variety of options, encompassing testing / profiling and run-time production features (the binary is under the gtfsproc directory).

  • -v, --version: Just display the version information.
  • -h, --help: Displays all command line options.
  • -d {path to feed}: File system location of the static feed (where stoptimes.txt, routes.txt, etc. are found).
  • -p {port number}: TCP port to open for client connections (defaults to 5000 if not specified).
  • -t {number of threads}: Specify the number of threads to dedicate to GtfsProc (this will equal the number of simultaneously-processed transactions, defaults to 1).
  • -r {URL or file path}: URL or local file path of the protocol buffer GTFS-Realtime file (optional).
  • -u {number of seconds}: Specify the number of seconds between each attempt to refresh GTFS-Realtime data (defaults to 120 seconds).
  • -a: Shows all times using a 12-hour format with AM/PM indicators (the default display is with 24-hour clocks).
  • -l: Looseness parameter for enforcing the start date/time values provided by real-time trip updates. (0 is the default and rigorously enforces the start date/time to match the service date, 1 ensures that the start date/time matches the actual date which helps for agencies who fill the date with the actual date and not the service date, and 2 provides no matching at all for agencies which provide no start date). Loosening the date enforcement (especially option 1) may lead to bugs when a trip is started late from a scheduled time on the previous day. Option 2 makes it so multiple trips can potentially be flagged with the same time and should not be used for agencies providing POSIX-style seconds-since-epoch trip updates. Option 2 also has the side effect of marking trips that are also not really operating but merely are the same trip IDs for a subsequent service date.
  • -q: Disables strict stop ID and stop sequence checking of real-time updates. This is useful for agencies which add extra stops in real-time trip updates that do not line up with the static schedule. Even with this option, these trips are still counted as mismatching for the sake of RTI and RPS transactions (for statistical awareness of the data condition).
  • -i: Displays a record of all transactions and data refetches to standard error.
  • -s: Limit of the number of trips that will appear in output per route of NEX module responses.
  • -m: Do not show terminating trips in NEX and NCF module responses (this is basically what Google Maps does).
  • -z {z-option list (comma-separate)}: processing override options (Z-Options). A list of supported Z-Options is provided herein.

The following options are only for debugging purposes

  • -f {year,month,day,hour,minute,second}: Force the time of all transactions to a fixed date and time (this is useful for debugging, in particular with integrating GTFS-Realtime information) where a naturally-advancing clock might make a condition/transaction unreproducible.
  • -x: Dumps the GTFS-Realtime decoded into a JSON format to standard error.

Simple Server Startup Guide

Looking to just start a server quickly? Here is an example workflow

  1. Find your data source online. A site like OpenMobilityData archives known transit agency data repositories.
  2. mkdir -p agency_data/MBTA && cd agency_data/MBTA
  3. wget https://cdn.mbta.com/MBTA_GTFS.zip
  4. unzip MBTA_GTFS.zip
  5. gtfsproc -d agency_data/MBTA -r https://cdn.mbta.com/realtime/TripUpdates.pb -u 90 -p 5000 -t 5
    This starts GtfsProc with the data downloaded in the MBTA, refetches realtime trip update data every 90 seconds, has 5 worker threads to handle simultaneous transactions, and answers clients connecting to TCP port 5000. All the static transit data is loaded into an in-memory database. See the example output below!

Automating the Data Retrieval and Server Restarts

In the real world, agencies will periodically update transit data at an unknown schedule. The provided GtfsProc_Agent.pl script can be updated per its instructions to stop the server, look for new data (downloading it if necessary) and reload the server automatically at an hour of your choosing each night.

Holding the Current Date for the GtfsProc system for testing

When testing GTFS-Realtime data with a static schedule, having a live system with a naturally advancing clock and live updates can make reproducing a bug extremely difficult. Let's be honest: there WILL be bugs, live data is a fickle beast and capturing the exact conditions to reproduce said bugs makes investigation easier! To remedy this, the "-f" option can be used to force the entire GtfsProc instance to an exact time and date. For example: "-f 2020,5,4,17,55,0" will freeze GtfsProc at 04-May-2020 17:55:00 (5:55:00 pm) in the local time of the agency in the data feed. When used in combination with a locally cached real-time Protocol Buffer trip update file, this is a very handy way to give time to investigate a bug. Here is an example of how to use it:

  • Say that you found a real-time trip update for a trip starting after midnight on a NEX request at 05-May-2020 at 00:35:09 EDT which does not get shown.
  • Capture the Protocol Buffer file, let's take RIPTA for example:
    wget "http://realtime.ripta.com:81/api/tripupdates?format=gtfs.proto -O ripta_tripupdates.pb"
  • Run GtfsProc with the local file and a fixed date (that you observed the issue):
    gtfsproc -d /opt/gtfsproc/RIPTA -r ripta_tripupdates.pb -f 2020,5,5,0,35,9
  • Run your query or queries, to see that the issue persists in the cached file
  • Try using DRT from a client to decode the protocol buffer file into a human readable form
  • Try using RTF or RTS for the trip ID in question to see what is available to GtfsProc from this trip.
  • Rerun GtfsProc with the same command, but with any overrides/parameters (usually the issues encountered are about date enforcement (use -l 1 or -l 2))
  • If you are really stumped, send me a mail with:
    • the agency static zip file (or a link to it if it's big),
    • copy of the protocol buffer with the real time trip updates,
    • the GtfsProc server start command used (including the -f option for the correct time!),
    • the modules/commands to reproduce
    • a description of the issue you are encountering.
  • NOTE: For the example here, I noticed that RIPTA does not enforce the service date (04-May-2020) in the real-time-trip update, instead opting to use the real start date (05-May-2020). Because of this, I know that the rigorous standard date checking of real-time trip updates is not wise to use for RIPTA, so I run GtfsProc with "-l 1" for handling RIPTA schedules.

Requesting and Viewing Data with a Client

All requests for wait times, schedules, stops, etc. are made from clients using transaction codes and (if necessary) additional information. See the Modules section below for details on each of these messages. The responses are encoded in JavaScript Object Notion (JSON) as that seems to be all the rage these days.

Included Command-Line Client

A rudimentary client command-line interface is included (the binary is in the client_cli directory). This tool allows you to send commands to the processor and then waits for the full JSON-encoded response (the end of a GTFSProc response is always indicated by a newline ‘\n’ character) to render the received data in a tabular format. Here is how to use it:

$ client_cli localhost 5000

This will connect to a server on localhost which receives requests on TCP port 5000. Upon startup, a small help window will display the supported transactions. If you resize the terminal window at all, the ncurses-based window geometry detection must be refreshed / "reinitialized" using "HOM" again.

Commands are entered at the ‘%’ prompt, but not all GtfsProc modules are supported by this interface. For how to enter commands, see the “Modules” section below. Override Options (Z)

The GtfsProc server may be started with optional process flags that override the default processing abilities when processing transations. These are typically for processing data that may not conform to the GTFS specification exactly. These are inherently fluid and will likely be added to through future versions of GtfsProc. The supported versions are provided below.

ALL_SKIPPED_IS_CANCELED

When ingesting a real-time trip update (RTTU) ProtoBuf into GtfsProc's internal data structures, if all stop time updates in a given trip update are listed as "canceled", then instead of marking that trip as skipping those stops, just say the trip is canceled. This operation will apply without considering the trip start date field if it is not filled in from the provider (so all future and past instances would also be marked as canceled). This is a work-around in place for the MBTA way of marking most bus trips as skipped when they're actually canceling the entire trip.

Modules

There are 18 modules available for integrating schedule/stop information and retrieving them over a TCP socket. All responses from GtfsProc are encoded in JSON (JavaScript Object Notation) an terminated with a newline ‘\n’ character.

Standard Response Parameters

All module responses are encoded in JSON.

If a request is received for a module (3-letter code) that does not exist, the standard response is: “{"error":1,"user_string":"(your bad request)"}”.

If the unthinkable (an exception) is raised, the standard response is “{"error":2,"user_string":"(your original request)"}”.

These above two conditions should be checked against. Successful responses will always have an error code of 0 along with the 3-letter code of the module you requested. All other error codes are module-specific.

All GtfsProc modules have the following parameters in addition to the module-specific responses:

Field Type Description
error integer Always a 0 for a successful transaction (individual modules have their own specific error codes)
message_type string Module name (3 letter code) of the response type
proc_time_ms integer Number of milliseconds it took to process the request (does not include any time in the queue waiting for an available thread)
message_time string Time at which the message began processing by a thread (format is either: “dd-MMM-yyyy hh:mm:ss z” or 12-hr format with AM/PM)

Real-Time Trip Information (RTI)

This is a simple module for showing: a) scheduled trips which contain real-time prediction information, b) all canceled trips, c) any trips that are supplemental to the static schedule.

Request Format

RTI

Response Format

This is purely for debugging / saving information for bug-reporting purposes and not for general consumption, so the response details will not be enumerated here. See the GTFS-Realtime specification for details about the feed format. This is the only GtfsProc module which does not obey the standard response fields.

Field Type Description
active_trips array List of all routes with scheduled trips with real-time predictions.
canceled_trips array List of all routes with at least one canceled trip.
added_trips array List of all routes with at least one added / supplemental trip.
orphaned_trips array List of all trip IDs which could not be associated with a route. For example: "orphaned_trips":["2135845","2135849"]
mismatch_trips object List of all routes with at least one real-time trip that matches a scheduled trip but contains unexpected sequences or stop ids.
duplicate_trips object List of trip IDs organized by route which are represented more than once in the real time feed, and within each list, the list of real-time trip update indexes (which can be directly queried with the RTT module).

The “active_trips”, “cancelled_trips”, “added_trips”, and "mismatch_trips" contain named objects (one for each route ID) with an array of Trip IDs, for example:
{"active_trips":{"Green-C":["43832582","43832587"]}}

The "duplicate_trips" set contains route IDs with an inner group of trip IDs with ye another inner group of real-time trip update indexes which are duplicates (of the same trip ID), for example:
"duplicate_trips":{"4150":{"357786731":[20,29]}}

Realtime Product Status (RPS)

This modules renders the status of GtfsProc's main backend (number of transactions, system time, uptime) as well as a full statistics diagnostic of the realtime data including statistics (download time, integration time, version) plus, for each route (and overall) the number of duplicate trips, mismatched trips, scheduled-and-running trips, added trips, and canceled trips. At a global level, any orphaned trips are enumerated. The idea for this is that it allows for a quick view of the status of all the trip updates acquired and any problematic conditions therein.

Request Format

RPS

Response Format

The "error" field will always be 0 for this transaction. The "message_type" is RPS. The full contents are detailed in the table below:

Field Type Description
uptm_s integer The number of seconds the backend has been running (clients should render this a little nicer in days/hours/minutes/etc.)
statdat string Modification time of the static dataset. If running a client long term, it's suggested to cache this parameter and check messages against it. New datasets could contain new route information which should be re-fetched.
nb_reqs integer Number of transactions the backend has handled since it was started.
ltst_rt string The most recent real-time-requiring transaction (like RTI, NEX, NCF, RTT/RTF/RTS)
rt_buff string "A" or "B" for an active buffer, "IDLE" if the backend had no real-time transactions within the last 3 minutes so it stopped fetching, "N/A" if data retrieval failed or the backend was not started with real-time data requested.
datagen string Data generation time taken directly from the agency's real-time trip update buffer. Will be "-" if unabailable.

gtfsrtv string The version string from the real-time trip update buffer.

Field Type Description

Request Format

    Upcoming / Next Service for Stop ID(s)

    The primary purpose of GtfsProc was envisioned for this purpose: displaying the wait time for upcoming service at a particular stop, list of stops, or parent station. The NEX/NCF modules accomplish this by loading all relevant trips* for the number of future minutes requested. If real-time data is relevant on the server, then the scheduled trips operating, added service, and canceled trip IDs are all merged and then for all trips who have no departed a stop or have not yet arrived will be sorted by the wait time to get to the station (if no arrival time is available, we look at the departure time for the countdown).

    * a “relevant” trip could come from a previous service day (when trips operate: a) after midnight on a previous service date render on the current date, b) the current date (naturally), and c) the next date (since the look-ahead time might have to pull trips from tomorrow’s service date).

    Next Service for Stop ID(s) - Organized by Route ID (NEX)

    This variation sorts all the upcoming services by route then wait time.

    Request Format

    NEX {look-ahead minutes} {stopID}
    NEX {look-ahead minutes} {stopID1}|{stopID2}|...|{stopIDn}

    Response Format

    The ‘message_type’ is “NEX”.
    Possible error values:

    • 0: Success
    • 601: A requested stop ID does not exist

    Field Type Description
    stop_id string Stop ID of the stop requested, or the parent station, or a “|” delimited list of all stop IDs requested
    stop_name string Name of the Stop ID or parent station (if one) or a “|” delimited list of all stop names requested
    stop_desc string Description of the Stop ID or “Parent Station” if a parent station was requested, or “Multiple Stops” if a list of stops were requested
    static_data_modif string Last-modified time and date of the GTFS Static Dataset for the whole transit agency.
    realtime_age_sec integer Age of the currently-active real-time feed buffer (same value as the active_age_sec field from RDS), relative to the agency time (even if frozen at startup with -f).
    routes array List of all routes that serve requested Stop ID

    The “routes” array contains the following fields:

    Field Type Description
    route_id string Route ID for the trip IDs contained within trips
    trips array List of all trips in order of wait_time_sec that serve requested Stop ID

    The “trips” array contains the following fields:

    Field Type Description
    trip_id string Unique identifier of the trip serving the stop
    short_name string Short name of the trip
    dep_time string Trip scheduled departure time from the stop ID (format is: “ddd hh:mm” for 24-hour format or with AM/PM indicated if started with 12-hour formats)
    arr_time string Trip scheduled arrival time at the stop ID (format is: “ddd hh:mm” for 24-hour format or with AM/PM indicated if started with 12-hour formats)
    wait_time_sec integer The number of seconds until the trip arrives at the stop ID. If only departure time is available, then it is the wait time for the departure. Note that for trips with no scheduled arrival/departure times nor any predictions available, the wait time displayed is actually for the next available predicted time in the sequence of the trip.
    headsign string Display on the front of the vehicle operating on the trip (if the headsign is unique to this stop, it will be shown instead of the trip’s overall headsign)
    pickup_type integer 0/Empty = Regular drop-off, 1 = No drop-off, 2 = Call agency to arrange drop-off, 3 = Coordinate with operator
    drop_off_type integer 0/Empty = Regular pickup, 1 = No pickup, 2 = Call agency to arrange pickup, 3 = Arrange/Flag operator
    trip_begins Boolean True if this stop is the first stop of the trip
    trip_terminates Boolean True if this stop if the last stop of the trip
    realtime_data object Real-time information pertaining to this trip

    The “realtime_data” set contains the following fields:

    Field Type Description
    status string Four-character code indicating the status of the trip in relation to the requested stop ID:
    • ARRV: Vehicle is arriving at the stop (within 30 seconds, this is only possible if the trip has a real-time arrival time)
    • BRDG: Vehicle is stopped at the stop (system time is after arrival but before departure, real-time information for departure and arrival required)
    • DPRT: Vehicle has departed the stop (departure time required in the real-time feed, this status is only shown so long as the stop is still in the realtime feed)
    • RNNG: Trip has real-time information
    • SKIP: Vehicle will skip the requested stop
    • CNCL: The trip has been canceled
    actual_arrival string Predicted arrival at the stop (format is: “ddd hh:mm” for 24-hour format or with AM/PM indicated if started with 12-hour formats)
    actual_departure string Predicted departure from the stop (format is: “ddd hh:mm” for 24-hour format or with AM/PM indicated if started with 12-hour formats)
    offset_seconds integer How late (positive value) or early (negative value) the trip is going to be at the requested stop. This is only valid if stop_status is "FULL".
    vehicle string The transit agency’s identifier of the [first, in case of a train, for example] vehicle operating on this trip
    stop_status string Four-character code indicating the presence of real-time vs. scheduled data:
    • FULL: The wait_time_sec and offset_seconds are filled based on scheduled and real-time information for the stop ID.
    • SCHD: Real-time data is available for the trip at large, but not for this particular stop (usually the current time is too far in the future and the whole trip is not populated with predictions yet (use RTS {trip ID} or RTF {trip ID} to see). The wait_time_sec field is therefore based off the scheduled time available and offset_seconds should be ignored.
    • PRED: The wait_time_sec field has been filled based off of prediction data, but there was no scheduled time from which to base offset_seconds (which should be ignored).
    • SPLM: The trip is supplemental (from the added_trips set) and therefore will no have schedule times from which to compute offset_seconds (which should be ignored).

    Trip Realtime Information for Route(s) (TRR)

    This module displays the list of all added (supplemental) trips and scheduled trips with real time data for the requested routes. This information includes vehicle numbers (if available), headsign (or a stop-specific one) from the static feed, or a guessed one based on added trip last-available-stop details, the first stop in the prediction feed along with the time of arrival and departure at that stop. Note that real time trip updates lacking trip start date / post-24-hour clocks will render the day of arrival incorrectly (potentially) ... effort was made to detect these situations and simply avoid printing the day-of-week along with the time to not look as bad. Many agencies also use improper start dates and times on real-time-trip-updates (not computed by the operating day but the actual local day that the trip started if it's after midnight). You've been warned. This is more of a module for entertainment purposes and tracking which vehicles are operating on a Route ID.
